# Recipe Taste Optimizer 프로젝트 발표 자료

## 발표 개요

- 대상: 프로젝트팀, 비전공자 포함
- 시간: 15~20분
- 목표: 서비스 목적, 핵심 기능, 구조, 검증 결과, 남은 과제를 팀 전체가 같은 수준으로 이해한다.
- 발표 톤: 기술 설명은 최소화하고 사용자 흐름과 운영 관점 중심으로 설명한다.

## 1. PPT 구성안

### 1) 제목

**Recipe Taste Optimizer 프로젝트 결과 공유**

- 레시피 탐색, AI 개량 요청, 관리자 검수까지 연결하는 WIZ 기반 웹 서비스
- 발표자 / 날짜

발표 시간: 1분

### 2) 프로젝트가 해결하려는 문제

핵심 메시지:

- 사용자는 상황에 맞는 레시피를 빠르게 찾고 싶다.
- 레시피 정보는 출처, 버전, 피드백 관리가 필요하다.
- AI가 제안한 레시피는 바로 공개하기보다 검수와 기록이 필요하다.

슬라이드 문구:

- 레시피 탐색: 검색, 추천, 상세 정보 확인
- 맞춤 개량: 저염식, 이유식 등 목적 기반 AI 요청
- 운영 관리: 출처, 레시피, 피드백, 로그를 관리자 화면에서 추적

발표 시간: 2분

### 3) 전체 사용자 흐름

핵심 메시지:

- 일반 사용자는 레시피를 찾고, 저장하고, 의견을 남긴다.
- 프리미엄/로그인 사용자는 AI 개량 요청을 할 수 있다.
- 관리자는 레시피와 AI 결과를 검수하고 서비스 품질을 관리한다.

슬라이드 문구:

1. 메인 화면에서 추천/인기 레시피 확인
2. 목록 화면에서 조건별 검색
3. 상세 화면에서 버전, 원본 출처, 댓글, 수정 요청 확인
4. AI 개량 요청 저장
5. 관리자 검수 후 공개 여부 결정

발표 시간: 2분

### 4) 주요 화면

핵심 메시지:

- 사용자는 서비스 첫 화면에서 바로 레시피를 탐색할 수 있다.
- 상세 화면은 레시피 정보와 신뢰 정보를 함께 보여준다.
- 관리자는 운영에 필요한 업무를 별도 메뉴에서 처리한다.

슬라이드 문구:

- 메인: 추천/인기 레시피
- 목록: 검색, 필터, 프리셋
- 상세: 재료, 조리 단계, 영양 정보, 출처, 댓글
- AI 개량 요청: 목적 선택, 조건 입력, 검수 대기 저장
- 관리자: 레시피, 출처, AI, 피드백, 시스템 로그

발표 시간: 3분

### 5) 데이터와 검수 구조

핵심 메시지:

- 레시피는 단일 글이 아니라 여러 버전과 출처를 가진 데이터로 관리한다.
- AI 결과는 사람이 검토할 수 있도록 요청/결과/승인 상태를 분리했다.

슬라이드 문구:

- 레시피 기본 정보: 이름, 설명, 태그, 상태
- 버전 정보: 재료, 조리 단계, 영양 정보, 변경 요약
- 출처 정보: 원본 링크, 수집 상태, 원문 보관 정책
- 사용자 활동: 댓글, 즐겨찾기, 최근 조회, 수정 요청
- 관리자 기록: 누가, 언제, 무엇을 처리했는지 로그화

발표 시간: 2분

### 6) 기술 구조 요약

핵심 메시지:

- 화면은 Angular/WIZ 앱 구조로 구성되어 있다.

## 2. 발표 대본

### 슬라이드 1. 제목

안녕하세요. 오늘은 Recipe Taste Optimizer 프로젝트의 현재 결과를 공유드리겠습니다.

이 프로젝트는 레시피를 단순히 보여주는 서비스가 아니라, 사용자가 목적에 맞는 레시피를 찾고, 필요하면 AI 개량 요청을 남기고, 관리자가 그 결과를 검수해서 운영할 수 있도록 만든 웹 서비스입니다.

오늘 발표는 기능 소개, 전체 흐름, 기술 구조, 이번 오류 점검 결과, 그리고 남은 과제 순서로 진행하겠습니다.

### 슬라이드 2. 프로젝트가 해결하려는 문제

먼저 이 프로젝트가 해결하려는 문제입니다.

레시피 서비스에서 사용자는 빠르게 원하는 음식을 찾고 싶어 합니다. 예를 들면 저염식이 필요하거나, 아이를 위한 이유식처럼 조건이 있는 경우입니다.

또 운영 입장에서는 레시피의 출처가 어디인지, 어떤 버전이 공개되어 있는지, 사용자가 어떤 피드백을 남겼는지를 관리해야 합니다.

여기에 AI가 들어가면 한 가지가 더 중요해집니다. AI가 만든 결과를 바로 공개하지 않고, 관리자가 검토하고 기록을 남길 수 있어야 합니다. 그래서 이 프로젝트는 사용자 편의성과 운영 검수 구조를 함께 잡는 방향으로 설계했습니다.

### 슬라이드 3. 전체 사용자 흐름

전체 흐름은 크게 세 단계입니다.

첫 번째는 사용자가 레시피를 찾는 단계입니다. 메인 화면에서 추천이나 인기 레시피를 보고, 목록 화면에서 검색이나 필터를 사용합니다.

두 번째는 상세 확인 단계입니다. 사용자는 레시피 상세에서 재료, 조리 단계, 영양 정보, 원본 출처, 댓글을 확인할 수 있습니다.

세 번째는 AI 개량과 관리자 검수입니다. 사용자가 특정 목적에 맞게 레시피 개량을 요청하면, 결과가 바로 공개되는 것이 아니라 검수 대기 상태로 저장됩니다. 관리자는 이를 확인하고 공개 여부를 결정할 수 있습니다.

### 슬라이드 4. 주요 화면

주요 화면은 사용자 화면과 관리자 화면으로 나눌 수 있습니다.

사용자 화면에는 메인, 레시피 목록, 레시피 상세, AI 개량 요청, 마이페이지가 있습니다. 사용자는 여기서 탐색, 상세 확인, 댓글, 수정 요청, AI 개량 요청을 처리합니다.

관리자 화면에는 레시피 관리, 출처 관리, AI 관리, 피드백 관리, 시스템 로그가 있습니다. 운영자는 어떤 레시피가 공개되는지, 어떤 출처가 실패했는지, 어떤 요청이 들어왔는지 확인할 수 있습니다.

비전공자 관점에서는 이 구조를 “사용자가 쓰는 화면”과 “운영자가 관리하는 화면”으로 이해하시면 됩니다.

### 슬라이드 5. 데이터와 검수 구조

이 서비스에서 레시피는 단순한 게시글 하나가 아닙니다.

레시피 기본 정보가 있고, 그 아래에 버전 정보가 있습니다. 버전에는 재료, 조리 단계, 영양 정보, 변경 요약 등이 들어갑니다.

그리고 출처 정보도 별도로 관리합니다. 원본 링크, 수집 상태, 원문 보관 정책을 따로 둬서 레시피의 신뢰성과 운영 추적성을 확보합니다.

AI 개량 요청도 별도 데이터로 관리합니다. 사용자의 요청, AI 결과, 검수 상태가 분리되어 있기 때문에 나중에 어떤 요청이 왜 승인되었는지 추적할 수 있습니다.

### 슬라이드 6. 기술 구조 요약

기술 구조는 Angular와 WIZ 프로젝트 구조를 사용했습니다.

쉽게 말하면 화면을 담당하는 부분, 요청을 처리하는 부분, 데이터를 가져오는 부분을 나눠 둔 구조입니다.

화면은 `src/app` 아래의 page와 layout으로 구성되어 있고, 공통 기능이나 재사용 가능한 기능은 `src/portal` 패키지에 있습니다. 데이터 접근은 Struct 패턴을 사용해서 사용자, 레시피, 출처, 피드백 같은 영역을 분리했습니다.

이렇게 나눈 이유는 기능이 커졌을 때 한 파일에 모든 로직이 몰리지 않도록 하기 위해서입니다.

### 슬라이드 7. 이번 점검 및 수정 결과

이번에 전체 페이지 오류를 확인하면서 개발 빌드 기준으로 페이지 컴파일 오류를 해결했습니다.

주요 오류는 누락된 SCSS 파일, Angular strict 타입 검사와 WIZ 생성 코드의 충돌, 전역 라우트 정보인 `WizRoute` 타입 문제, `wiz.call()` 응답 타입 문제 등이었습니다.

수정 후 `npm run build -- --configuration development` 명령으로 개발 빌드가 성공했습니다.

다만 production 빌드는 아직 실패합니다. 이 실패는 페이지 코드 오류가 아니라 번들 용량 제한 때문입니다. 현재 초기 번들 크기가 설정된 production budget보다 커서 실패하는 상태입니다. 따라서 남은 TODO는 번들 용량 최적화 또는 budget 기준 재검토입니다.

### 슬라이드 8. 현재 완성도와 한계

현재 MVP 기능 흐름은 갖춰져 있습니다.

사용자는 레시피를 탐색하고 상세를 확인할 수 있고, AI 개량 요청을 남길 수 있습니다. 관리자는 레시피, 출처, AI, 피드백, 로그를 관리할 수 있습니다.

하지만 운영 배포 전에는 추가 검증이 필요합니다. 개발 빌드 성공은 코드가 컴파일된다는 의미이고, 실제 사용자가 브라우저에서 모든 시나리오를 문제없이 수행한다는 뜻은 아닙니다.

그래서 다음 단계에서는 실제 사용자 시나리오 기반 브라우저 테스트, 관리자 검수 플로우 QA, 보안 취약점 확인, production build 최적화가 필요합니다.

### 슬라이드 9. 다음 단계

다음 단계는 기능 추가보다 출시 가능 품질 확보가 우선입니다.

첫 번째는 production build 통과입니다. 번들 용량을 줄이거나 budget 기준을 현실적으로 조정해야 합니다.

두 번째는 주요 사용자 시나리오 테스트입니다. 메인, 목록, 상세, AI 요청, 로그인, 마이페이지, 관리자 화면을 실제 흐름대로 확인해야 합니다.

세 번째는 보안과 운영 점검입니다. npm audit에서 나온 취약점의 영향도를 확인하고, 배포와 롤백 절차를 문서화해야 합니다.

### 슬라이드 10. 마무리

정리하면, Recipe Taste Optimizer는 레시피 탐색과 AI 개량 요청, 그리고 관리자 검수를 하나의 흐름으로 연결한 프로젝트입니다.

현재 개발 빌드 기준 페이지 오류는 해결했고, 남은 핵심 과제는 production build 최적화와 실제 브라우저 기반 QA입니다.

이후 작업은 기능을 더 늘리기보다, 현재 만들어진 흐름을 안정적으로 배포 가능한 수준까지 끌어올리는 데 집중하면 됩니다.

## 3. PPT 제작용 요약 문구

### 한 줄 소개

Recipe Taste Optimizer는 사용자의 식단 목적에 맞는 레시피 탐색과 AI 개량 요청, 관리자 검수 운영을 연결하는 WIZ 기반 레시피 서비스입니다.

### 핵심 가치

- 사용자: 원하는 조건의 레시피를 빠르게 찾는다.
- 운영자: 출처, 버전, 피드백, 로그를 관리한다.
- 팀: AI 결과를 검수 가능한 상태로 운영한다.

### 핵심 기능

- 레시피 추천/검색/상세
- 레시피 버전 및 원본 출처 관리
- 댓글, 즐겨찾기, 최근 조회
- AI 기반 레시피 개량 요청
- 관리자 레시피/출처/AI/피드백/로그 관리

### 검증 결과

- 개발 빌드 성공
- 페이지 컴파일 오류 해결
- production build는 번들 용량 제한으로 실패

## 4. 예상 질문과 답변

### Q1. 지금 바로 배포 가능한가요?

아직은 권장하지 않습니다. 개발 빌드는 성공했지만 production build가 번들 용량 제한으로 실패하고 있습니다. 운영 배포 전에는 production build 통과와 실제 브라우저 QA가 필요합니다.

### Q2. AI 결과는 자동으로 공개되나요?

아닙니다. AI 개량 요청은 검수 대기로 저장되는 구조입니다. 관리자가 결과를 확인하고 공개 여부를 결정하는 흐름을 전제로 설계했습니다.

### Q3. 비전공자가 이해할 때 이 프로젝트의 핵심은 무엇인가요?

사용자는 레시피를 찾고 요청을 남기고, 관리자는 그 요청과 데이터를 검수해서 서비스 품질을 관리하는 구조입니다. AI는 자동 공개 도구가 아니라 운영자가 검수할 수 있는 제안 도구로 쓰입니다.

### Q4. 이번에 해결한 오류는 어떤 의미인가요?

페이지가 개발 환경에서 컴파일되지 않던 문제를 해결했다는 의미입니다. 누락된 스타일 파일, 타입 선언 문제, 전역 라우트 정보 문제 등을 정리해서 개발 빌드가 성공하도록 만들었습니다.

### Q5. 남은 가장 큰 리스크는 무엇인가요?

가장 큰 리스크는 production build 최적화와 실제 사용 시나리오 검증입니다. 코드가 빌드되는 것과 사용자가 모든 화면을 문제없이 쓰는 것은 다르기 때문에 브라우저 기반 QA가 필요합니다.
